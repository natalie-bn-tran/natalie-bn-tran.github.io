Network Configuration

Completing the networking for each of the machines is crucial. This will give the machines their own IP address, allowing communication, data transportation, and sets the stage for changing network settings.
IP Address: A unique identifier for a device on a network.

To make things easier, the LAN subnet will be same as the one automatically set in the pfSense router (192.168.1.0/24), and the last octets of the non-router machines will be 2-10.

pfSense

WAN: It's set with DHCP, but it's recommended to set it statically.
LAN: This can be kept to the default IP address but can be changed as well.

Change the IP address of a pfSense router, using the CLI:

  1. Enter 2
  2. Enter 1 or 2, depending on if the WAN or LAN is being changed
  3. Enter the new IP address in this format: xxx.xxx.xxx.xxx/yy
    • Note: xxx.xxx.xxx.xxx is the IP address, and /yy is the subnet in CIDR format (the subnet can be entered after if it was not written with the IP address).
    • For WAN: yy = 16
    • For LAN: yy = 24
  4. For WAN: the default gateway is 172.16.0.1, and enter y when prompted if this should be the default upstream gateway
    • Note: This is only applicable if the SDC infrastructure is being used. 172.16.0.1 is the IP address of the SDC's router and will allow the VMs to have Internet access.
  5. Enter n or hit Enter for the rest of the options

For the lab, it is recommended to have 1:1 NAT set up on the router to have the ability to work from the host machine.
Network Address Translation (NAT): Translates IP addresses between interfaces.
1:1 NAT: Translates a WAN IP address to a LAN IP address and vice versa.

  • Having 1:1 NAT grants the ability to access the various services on a host machine instead of the VMs.
  • Ex: 172.16.187.0/24 would translate to 192.168.1.0/24, so 172.16.187.23 is 192.168.1.23.

Set up 1:1 NAT:

  1. On the host machine, open a browser and enter the pfSense's WAN IP address
  2. Login with the credentials admin : pfsense
  3. Create a 1:1 NAT firewall rule:
    1. Navigate to Firewall > NAT > 1:1
    2. Make the following selections:
      1. Interface: WAN
      2. Address Family: IPv4
      3. External subnet IP: WAN Address
      4. Internal IP: LAN subnet
      5. Destination: Any
      6. NAT reflection: Use system default
    3. Save and apply the changes
  4. Configure a virtual IP:
    1. Navigate to Firewall > Virtual IP
    2. Make the following selections:
      1. Type: Proxy ARP
      2. Interface: WAN
      3. Address type: Network
      4. Address(es): 172.16.xxx.0/24

        5. Note: Here, the subnet is /24 and not /16 because the only octets that are translated are the first three (network ID), and the last octet (host ID) needs to remain the same across the translation. /16 is for setting up the IP and making sure it's on the same subnet as the SDC router.

    3. Save and apply the changes

    Note: Replace 241 with the third octet of the WAN IP address.

Windows

The network configuration is the same for all of the Windows machines.

Change the IP address of a Windows machine:

  1. Run Win + R and enter ncpa.cpl or navigate to
    Control Panel > Network and Internet > Network and Sharing Center
  2. Select Ethernet > Properties > IPv4 > Properties
  3. Select Use the following IP Address and fill out the information:
    1. IP Address: 192.168.1.x
    2. Subnet Mask: 255.255.255.0 (automatically fills out when clicked)
    3. Default Gateway: {Router LAN IP}
    4. Preferred DNS server : 8.8.8.8 (Google's Public DNS)
  4. Select OK to apply the changes

    5. Note: Replace 192.168.1.5 with the appropriate IP address for each Windows machine and 192.168.1.1 with {Router LAN IP}.

Active Directory

Active Directory (AD): A giant database and central management for domain users, computers, and services.
There are two main components of AD: LDAP and DNS.
LDAP: Lightweight Directory Access Protocol - it puts the directory in Active Directory.
DNS: Domain Name System - allows LDAP to reference names in its directory.

In Evan's Lab:

  • Windows Server 2022: Primary domain controller (DC) and DNS server
  • Windows Server 2012 R2: Secondary DC and DNS server
Domain Controllers

There are two parts of setting up domain controllers: installing Active Directory Domain Services, which is the same for both DCs, and promoting the device to a DC.

Windows Server 2022: Primary DC

DNS settings: Change the DNS to 127.0.0.1 (localhost).
Localhost: The hostname of the machine (refers to itself).

Install AD:

  1. Navigate to Server Manager > Manage > Add Roles and Features or
    Control Panel > Programs > Programs and Features > Turn Windows Features On or Off
  2. Make the following selections and click next for other options:
    1. Installation Type: Role-Based
    2. Server Roles: Active Directory Domain Services
  3. Install
  4. Rename the device using PowerShell:
    5. Rename-Computer -NewName {name} -restart

Create a primary DC:

  1. Navigate to Server Manager > Flag > Promote to Domain Controller
  2. Make the following selections for the pop-up and click next for other options:
    1. Deployment Configuration: add a new forest and enter a domain name

      2. Note: Adding a new forest creates a new domain.

    2. Domain name: Enter a FQDN

      3. FQDN: Fully Qualified Domain Name.
      Ex: ejuan.detergent

    3. Domain Controller Options: Set the functional level and password

      4. Functional Level: Oldest version of Windows that any domain controller has; in this case, it would be Windows Server 2012 R2.

    4. Password: To make things easier, the password will be the same as the one set for the device
    5. Additional Options: ensure the domain is correct; it's one capitalized word

      6. Ex: EJUAN

    6. Prerequisites Check: Install
  3. Wait for the machine to restart
Windows Server 2012 R2: Secondary DC

DNS changes:

  1. Change the primary DNS server to the IP address of the primary DC

    2. Note: The Windows Server 2022 has the IP 192.168.1.2 and uses its localhost (itself) for the DNS server.
    In order for the Windows Server 2012 R2 to access the records on the domain, its preferred DNS server would be 192.168.1.2, which is where the primary DNS server is located.
    If the preferred DNS server was set to localhost for the Windows Server 2012 R2, it would try to access the domain's DNS records by making requests to itself, but it does not have them.

  2. Change the alternate DNS server to localhost

    3. Note: The Windows Server 2012 R2 has the secondary DNS server in the case that the primary DC's DNS does not work; it will be used as a backup.

Install AD the same way as with the Primary DC, rename the device, and promote to a DC.

Create a Secondary DC:

  1. Make the following selections for the pop-up and click next for other options:
    1. Deployment Configuration: Add a domain controller to an existing domain, enter the domain name, and enter the credentials
    2. After entering the domain name: click on its name and OK
    3. Credentials: Enter the credentials of the Administrator account on the domain

      4. Note: The username is in the format DOMAIN\Administrator.
      The password can be changed on the Primary DC if needed by navigating to
      Active Directory and Users > SLD.TLD > Users > right click Administrator > Reset Password

    4. Domain Controller Options: Add a password; everything else should be set to default

      5. Note: Ensure DNS server and DC are selected.

    5. Additional Options > Replicate From: Domain Name
    6. Prerequisites Check: Install
  2. Wait for the machine to restart

NGINX Web Proxy

Machine: Debian 11
Purpose: Proxy for the IIS website hosted on Windows Server 2012 R2

Create the IIS website on Windows Server 2012 R2:

  1. Navigate to Server Manager > Manage > Add Roles and Features
  2. Make the following selections for the pop-up and click next for other options:
    1. Installation Type: Role-Based
    2. Server Roles: Web Server (IIS)
  3. Install
  4. Run Win + R and enter inetmgr.exe
  5. Click on {Windows Server 2012 R2 Hostname} > View Sites > Add Website

    6. Note: The default website will be used. The website's port information is given.

  6. In the browser, navigate to http://{Windows Server 2012 R2 IP}:80

Install and enable NGINX:

  1. Install NGINX on Debian 11:
    2. apt install nginx -y
  2. Start and enable NGINX:
    3. systemctl start nginx
systemctl enable nginx
systemctl status nginx

Configure the reverse proxy:

  1. Unlink the default configuration file:
    2. unlink /etc/nginx/sites-enabled/default

    Note: The reverse needs a custom configuration file, and there can be conflicts with having two. The default configuration file will be kept as a backup.

  2. Create a new configuration file:
    3. touch /etc/nginx/sites-available/reverse-proxy

    Note: vim and nano can be used in place of touch to create and edit the file with one command.

  3. Add the following:
    4. server {
    listen 80;
    server_name localhost;
    location / {
        proxy_pass http://{Windows Server 2012 R2 IP}:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
  4. Link and activate the new configuration file:
    5. ln -s /etc/nginx/sites-available/reverse-proxy /etc/nginx/sites-enabled/
  5. Confirm the configuration is successful:
    6. nginx -t
  6. Restart NGINX:
    7. systemctl restart nginx

Test NGINX:

  1. On the host machine's browser, navigate to the external IP of the NGINX server

Security Considerations

This covers some things to think about in relation to security such as service configurations, file permissions, passwords, and firewalls.

Proprietary Data

Proprietary data: 🍆
The proprietary data will not be leaked. There was a domain user created for the SMB share, and only that user is able to access it.

HR Data

HR data: 🍑
The HR data was secured with SFTP. A new group, user, and password were created for SFTP. A chroot directory was made so the user is restricted to that directory, where they only have read permissions, meaning the SFTP user can only see items in the directory but can't make changes to it.

IIS Server

The IIS website should only be accessible through the reverse proxy. However, it can be accessed using {Windows Server 2012 R2 IP}. This can be changed with firewalls.
Firewall: A networking tool that manages network traffic.

Make sure the Windows Server 2012 R2 firewalls are on and check default policies (GUI):

  1. Run Win + R and enter wf.msc. The overview should show the firewall profiles are on:
  2. Click on Windows Defender Firewall Properties. For each profile, ensure the inbound connections are set to block and the outbound connections are set to allow

Add an inbound rule (GUI):

  1. Click on Inbound Rules > New Rule
  2. Make the following selections:
    1. Rule Type: Custom
    2. Program: All Programs
    3. Protocols and Ports > Protocol Type: TCP
    4. Protocols and Ports > Local Port: Specific Ports, 80
    5. Scope > Remote IP: {Ubuntu 22 IP}
    6. Action: Allow the Connection
    7. Profile: All
    8. Name: Enter a name for the rule
  3. Click Finish to complete creating a new rule

Check if the IIS site can still be reached using {Windows Server 2012 R2 IP}:

  1. On the host machine, navigate to the external IP of the Windows Server 2012 R2
  2. If it can be reached that way, there might be conflicting inbound firewall rules that affect connections to port 80
  3. Filter the inbound rules by local port: Scroll sideways and click on Local Port until the top ports start with RCP
  4. Scroll down, find active rules for port 80, and disable them
NSA's Beast

NSA created EternalBlue, a vulnerability found within SMBv1 that can be exploited to gain unauthorized access.

Preventing EternalBlue:

  1. Disable/remove SMBv1 (in PowerShell):
    2. Disable-WindowsOptionalFeature -Online -FeatureName SMB1Protocol
Set-SmbServerConfiguration -EnableSMB1Protocol $false
  2. Upgrade to a higher version of SMB

Requirement 1: “Install and maintain a firewall configuration to protect cardholder data”
In Evan's Lab: Add firewalls on the machines, not including the router, to only allow necessary traffic.
